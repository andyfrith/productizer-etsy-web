Constitution / Git Workflow

Git Workflow

All phase work lands on dedicated branches and merges into master via pull requests. No direct commits to master for features, specs, or scaffolding.

Decided
  • Integration branch: master
  • Phase branches: phase/{N}-{slug} (one branch per roadmap phase deliverable)
  • Merge path: PR only (review + validation gate before merge)

Branch naming

PatternWhenExample
phase/{N}-{slug} All work for roadmap phase N — specs, code, tests, README updates phase/0-scaffold, phase/1-concept-crud
fix/{short-slug} Urgent bugfix off master (not a full phase) fix/db-health-500
chore/{short-slug} Tooling/docs only, no product phase chore/ci-playwright

{N} = roadmap phase number (0, 1, 2…). {slug} = kebab-case from phase title (e.g. scaffold, concept-crud, asset-upload). Align with spec folder specs/{YYYYMMDD}-phase{N}-{slug}/ when present.

Lifecycle (per phase)

master (protected integration)
   │
   ├─ git fetch origin && git checkout master && git pull
   ├─ git checkout -b phase/N-slug
   │
   ├─ work: specs + implementation + tests
   ├─ commits on phase/N-slug only
   ├─ push: git push -u origin phase/N-slug
   │
   ├─ open PR → master
   ├─ validation gate (see phase validation.html)
   ├─ review / approve
   └─ merge PR → delete branch (optional)

Starting a phase branch

# From repo root — always branch from latest master
git fetch origin
git checkout master
git pull origin master

# P0 example
git checkout -b phase/0-scaffold

# Confirm
git branch --show-current   # → phase/0-scaffold

Pull request rules

  • One phase per PR — do not mix P0 scaffold with P1 concept CRUD in the same branch/PR.
  • Title: phase(N): short description — e.g. phase(0): scaffold Next.js app and dev stack
  • Body: Copy checklist from specs/…/validation.html; link phase spec folder.
  • Before open: Agent runs ./scripts/validate-pN.sh (exit 0); paste summary in PR; no secrets in diff.
  • After merge: Pull master locally before starting the next phase branch.

What not to do

  • Commit phase work directly on master
  • Long-lived personal branches that bundle multiple phases
  • Force-push to master
  • Merge without running the phase validation checklist

Agent / Cursor

Skills and rules enforce this workflow:

  • Skill: productizer-git-workflow
  • Rule: .cursor/rules/git-workflow.mdc (always apply)

Before creating or editing files for a phase, the agent must confirm the current branch is phase/N-slug or create/checkout that branch from master.
