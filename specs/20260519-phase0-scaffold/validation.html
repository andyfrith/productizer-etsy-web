Validation — P0 Scaffold

Agents run automation first. Use ./scripts/validate-p0.sh, curl, and Playwright. Manual user checks are minimal (merge approval, environment blockers only).

PR-ready when: ./scripts/validate-p0.sh exits 0, branch is phase/0-scaffold, agent pasted validation summary in PR. Skill: productizer-phase-validate.

1. Agent — run validation script

From repo root. Agent must execute and fix failures before opening/updating PR.

chmod +x scripts/validate-p0.sh scripts/validate-lib.sh

# Before Next.js exists (docs / infra only)
./scripts/validate-p0.sh --tier docs

# After scaffold complete (default)
./scripts/validate-p0.sh

Script prints PASS / FAIL per check and exits non-zero on failure.

3. Agent — curl reference (debug)

# Health (DB up)
curl -s http://localhost:3000/api/health/db | jq .

# Home
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://localhost:3000/

# DB down test (script does this automatically)
docker compose stop postgres
curl -s http://localhost:3000/api/health/db | jq .
docker compose up -d postgres

4. Playwright — UX automation (required in e2e)

Encode former “manual UX” items as E2E assertions so validate-p0.sh covers them:

  • Title / branding: Productizer in <title> or heading
  • Studio shell: not default Next.js welcome page
  • DB status element visible (data-testid="db-status" recommended)
  • Skeleton or loading state during fetch
  • At least one focusable control (button/link)

5. User — only when blocked

  • Docker Desktop not running — start Docker, re-run script
  • Port 3000/5432 in use — free port or change compose port
  • GitHub: approve and merge PR
