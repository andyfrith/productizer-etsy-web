Requirements — P3 Variation Lineage

Functional and technical requirements for variation branching and approval. Satisfies roadmap phase 3 and unblocks P4 AI generation.

Locked decisions
  • Phase: P3 — full roadmap variation lineage scope
  • API: REST under /api/concepts/:id/variations
  • Runtime: pnpm, Node 22 LTS
  • Validation bar: automated script + curl + Playwright approve E2E
  • Spec folder: specs/20260520-phase3-variation-lineage/
  • Git branch: phase/3-variation-lineage → merge via PR to master

Goal

On a concept detail page, a user can create variations (uploaded art directions) with optional parent/child lineage, compare them in a gallery, and approve or reject candidates. At most one variation is approved per concept at a time. Variations are distinct from P2 reference assets but may be seeded from a reference image.

Prerequisites (P2 merged)

  • design_concepts + design_assets; preview_asset_id; reference gallery APIs
  • sharp rendition pipeline under STORAGE_ROOT (reuse patterns from P2)
  • P2 validation: ./scripts/validate-p2.sh passes on master

Requirements (MUST)

IDRequirementAcceptance
R3.1 concept_variations table Drizzle schema + migration: id (uuid), concept_id (FK → design_concepts, cascade delete), parent_variation_id (nullable self-FK), source_asset_id (nullable FK → design_assets), status (pending | approved | rejected), prompt_snapshot (nullable text, for P4), storage_key, thumb_key, card_key, full_key, byte_size, optional width/height, created_at, updated_at; aligns with tech-stack excerpt
R3.2 Approved variation link design_concepts.approved_variation_id (nullable FK → concept_variations.id); set when a variation is approved; cleared when that variation is rejected or deleted
R3.3 List variations API GET /api/concepts/:id/variations returns JSON array sorted by createdAt desc; each item includes id, conceptId, parentVariationId, sourceAssetId, status, thumb URL or path helper, createdAt; 404 if concept missing
R3.4 Create variation (upload) POST /api/concepts/:id/variations accepts multipart/form-data: field file (required), optional parentVariationId; same MIME/size rules as P2 assets; writes renditions under {STORAGE_ROOT}/variations/{id}/; returns 201 + variation JSON
R3.5 Create variation from reference POST /api/concepts/:id/variations/from-asset JSON body { assetId, parentVariationId? }; copies rendition files from the reference asset into variation storage (server-side); sets source_asset_id; 400 if asset not linked to concept or archived
R3.6 Update variation status PATCH /api/concepts/:id/variations/:variationId body { status } where status is pending | approved | rejected; validates concept ownership
R3.7 Single approved rule When status becomes approved, any other approved variation for the same concept becomes pending (or previous approved cleared from approved_variation_id); approved_variation_id on concept updated to the new winner
R3.8 Serve variation images GET /api/variations/:id/file?variant=thumb|card|full streams bytes with correct Content-Type; 404 when missing
R3.9 VariationGallery UI On concept detail: grid of variation cards with thumb, status badge (pending/approved/rejected), parent indicator when parentVariationId set; data-testid="variation-gallery"
R3.10 Approve / reject actions Each pending card: Approve (data-testid="variation-approve") and Reject (data-testid="variation-reject"); approved card shows distinct styling; Sonner toast on success
R3.11 Upload & branch UX data-testid="variation-upload" file input + submit; optional Branch from preview (data-testid="variation-from-preview") when concept has previewAssetId; optional Branch on a card sets parentVariationId for next upload
R3.12 Framer Motion Gallery uses framer-motion staggered entrance for cards (staggerChildren on container); respects prefers-reduced-motion (instant layout, no stagger)
R3.13 TanStack Query hooks useVariations(conceptId), mutations for create, create-from-asset, patch status; invalidates ['variations', conceptId] and concept detail on success
R3.14 Unit tests Vitest: variation Zod schema and/or status transition helper (single-approved rule)
R3.15 E2E approve flow Playwright e2e/variation-lineage.spec.ts: create concept, upload variation (or from preview), approve one, assert approved badge and second approve demotes first
R3.16 Validation script ./scripts/validate-p3.sh with --tier docs before implementation; full tier before PR

  • Studio dashboard hero + variation count home page (P6)
  • Auth, multi-user, Etsy, niche research
  • Hard-delete variations (archive pattern deferred; reject status suffices for P3)
