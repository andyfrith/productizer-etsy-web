Requirements — P2 Asset Upload

Functional and technical requirements for reference art uploads. Satisfies roadmap phase 2 and unblocks P3 variation lineage.

Locked decisions
  • Phase: P2 — full roadmap asset upload (thumb, card, full)
  • API: REST — POST /api/assets (multipart), GET /api/assets/:id, file route with ?variant=
  • Runtime: pnpm, Node 22 LTS
  • Validation bar: automated script + curl + Playwright upload E2E
  • Spec folder: specs/20260520-phase2-asset-upload/
  • Git branch: phase/2-asset-upload → merge via PR to master

Goal

A user can upload multiple reference images for a design concept from the concept detail page. The server stores three renditions per image under STORAGE_ROOT and records metadata in Postgres. The user chooses one reference as the preview image (gallery thumb and primary tile).

Prerequisites (P1 merged)

  • design_concepts table and CRUD API/UI
  • src/lib/storage.ts — ensureStorageRoot() (P0)
  • STORAGE_ROOT in .env.example; storage/ gitignored
  • P1 validation: ./scripts/validate-p1.sh passes on master

Requirements (MUST)

IDRequirementAcceptance
R2.1 design_assets table Drizzle schema + migration: id (uuid), concept_id (uuid, nullable FK → design_concepts), original_filename, mime_type, storage_key (base path under storage root), thumb_key, card_key, full_key, byte_size, optional width/height, created_at
R2.2 Concept reference link design_concepts.preview_asset_id (uuid, nullable FK → design_assets.id); user-selected preview among references; first upload auto-sets preview when none
R2.3 Upload API POST /api/assets accepts multipart/form-data: field file (required), optional conceptId; validates MIME (image/png, image/jpeg, image/webp) and max size (e.g. 10 MB); returns 201 + asset JSON (camelCase)
R2.4 Rendition generation On upload, write files under {STORAGE_ROOT}/assets/{assetId}/: thumb (max edge ~256px), card (~800px), full (original or max edge ~2048px); use sharp (or equivalent) server-side
R2.5 Asset read API GET /api/assets/:id returns metadata; GET /api/assets/:id/file?variant=thumb|card|full streams bytes with correct Content-Type; 404 when missing
R2.6 Storage safety Keys are server-generated (no user path segments); reject path traversal; storage root from STORAGE_ROOT only
R2.7 Upload UI Concept detail: reference gallery (GET /api/concepts/:id/references) with multiple saved tiles; one marked Preview (asset-preview, primary ring); others asset-reference-card with Set as preview (PUT /api/concepts/:id/preview); pending local tile (dashed amber); remove pending (local undo) or saved (deferred delete + undo); upload adds to library without replacing others
R2.8 Client hook useUploadAsset mutation (FormData POST); invalidates concept detail query on success
R2.9 Dev tooling pnpm db:studio script documented in README for browsing tables
R2.10 Unit tests Vitest: upload validation schema and/or storage path helpers
R2.11 E2E upload Playwright e2e/asset-upload.spec.ts: upload flow; remove pending/saved with Undo toast restoring preview
R2.12 P1 regression Existing concept CRUD and P0 health checks still pass in validate-p2.sh gate

Out of scope

  • concept_variations table and variation gallery (P3)
  • AI image generation (P4)
  • Poster mock / product types (P5–P6)
  • Cloud storage adapter S3/R2 (P16)
  • Auth, multi-user asset ACLs
  • Video or SVG uploads
  • Bulk asset library page (deferred)
