Constitution / Validation Policy

Validation Policy

Every phase spec must be verifiable by an agent with minimal human intervention. Prefer scripts, curl, and Playwright over manual checklists.

Hierarchy

  1. Shell script — scripts/validate-pN.sh (single command, exit code)
  2. curl / HTTP — APIs and HTML smoke
  3. Playwright — UX, a11y, branding (not human eyes)
  4. User — only environment blockers + GitHub merge

Phase validation.html template

  • Lead with agent script command
  • Table: check → automated method
  • “Manual” section ≤ 3 items (Docker down, ports, merge)
  • PR template includes pasted script output

In-phase amendments (drop-in requests)

While a roadmap phase is active on phase/{N}-{slug}, the user may request small additions or UX fixes that stay within that phase’s scope. Treat these as spec + code work—not silent code-only changes.

  1. Confirm the request fits the current phase (not the next roadmap row). If it belongs to a later phase, say so and defer.
  2. Update the active phase folder specs/{YYYYMMDD}-phase{N}-{slug}/:
    • requirements.html — adjust acceptance criteria; add a row to Phase amendments
    • validation.html — new or changed automated checks (Playwright data-testid, curl, script)
    • plan.html — note under In-phase drop-in requests when helpful
    • index.html — one-line summary under In-phase updates when user-facing
  3. Update the phase implementation skill (e.g. productizer-p2-asset-upload) if behavior is non-obvious for future agents.
  4. Implement on the same phase branch; run ./scripts/validate-pN.sh before PR.
  5. README: only if the drop-in changes developer or user-facing workflows.

Enforced by productizer-git-workflow, productizer-phase-spec, and .cursor/rules/git-workflow.mdc.

Skills
