Brainstorm / Documentation style

Documentation style

How we write constitution and brainstorm pages so both people and agents can parse intent quickly—without decorative icons or ambiguous graphics.

Principle
Content first: titles, lead paragraphs, tables, and lists carry meaning
For LLM agents
Each article starts with a doc-context block (purpose, audience, status)
For humans
Light background, comfortable line length, semantic callouts (decided / deferred / focus)

What we avoid in docs

  • Abstract “mini mock” shapes that do not map to a specific UI screenshot
  • Emoji or icon grids that restate what the heading already says
  • Dark themes with high-contrast accent colors that distract from reading
  • Flows drawn only as nameless boxes without labels

What we use instead

NeedPatternExample
Product types .chip / .chip-list Poster, Sign, Tumbler
Phase or lane .lane-tag, .phase-tag now, next, later
Sequence pre.text-flow or roadmap table Concept → Variations → Mock
Locked choice .decided Wave 1 POD scope
Out of scope .deferred Niche research module
Current priority .focus-now P0–P6 concept studio

App UI vs documentation

The running Productizer app uses real thumbnails, mock composites, and galleries where visuals help sellers decide. Specification HTML stays textual so agents can grep paths, phase IDs, and acceptance criteria without interpreting decorative CSS art.

Stylesheet

Shared styles: specs/shared.css (imported by brainstorm/shared.css). Palette: warm off-white background, forest-green links, slate body text.
